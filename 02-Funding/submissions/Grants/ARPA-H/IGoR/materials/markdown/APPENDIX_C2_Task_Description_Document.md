# Appendix C.2: Task Description Document (TDD)

> [!NOTE]
> **Type:** DOCX · **Source file:** `materials/APPENDIX+C.2_Task+Description+Document_IGoR.docx` · **Extracted text:** `materials/extracted-text/APPENDIX_C.2_Task_Description_Document_IGoR.txt`
> **Governs:** The structured task-level specification for all proposed work; used as the cost cross-walk and milestone accountability document.

## At a glance (BLUF)

The TDD is the detailed task ledger for the IGoR proposal. It has no page limit and does not count against the 40-page cap. Every task and subtask must map to a specific Phase and Technical Area, include quantitative milestones, and be cross-walked to the Cost Proposal (C.3). The TDD drives post-award milestone acceptance and payment justification.

## Key facts

| Item | Requirement |
|---|---|
| **Page limit** | None; does not count against the 40-page Technical Proposal cap |
| **Task structure** | Organized by Phase and Technical Area |
| **Milestone specificity** | Quantitative performance criteria required (not qualitative statements) |
| **Cost cross-walk** | Tasks and milestones must map to Cost Proposal (C.3) |
| **Human/Animal subjects disclosure** | Required at the task level for each applicable task |
| **Deliverable IP assertion** | Required for any deliverable with less than Unlimited Rights |
| **Proprietary information** | Must not be included in the TDD |

## Structure / contents

1. **Effort summary**, general objective and complete task/subtask list organized by Phase and Technical Area

2. **Task list outline** (hierarchical numbering example from the template):
   - Phase: #
     - 1.1.0 Task Title
       - 1.1.1 Subtask Title
       - 1.1.2 Subtask Title
       - 1.1.3 Subtask Title
     - 1.2.0 Task Title
       - 1.2.1 Subtask Title
       - 1.2.2 Subtask Title
       - 1.2.3 Subtask Title
     - (Repeat as needed)

3. **Per-task tables** (one per task/subtask; add tables as needed):
   - Header row: Phase number, Task number, Task title
   - Task Objective and Task Description / Detailed approach
   - Location (where the work is performed)
   - Organization responsible for the task
   - Deliverables (with IP assertion if less than Unlimited Rights)
   - Human Subjects and/or Animal Subject Research (Yes/No; describe if yes)
   - Associated Milestones (observable/measurable events; at least three milestone rows provided; add/delete as needed)

## What it requires of a proposer

- **Summarize the effort and its importance** in the opening section before the task list.
- **Number tasks hierarchically** by Phase and Technical Area (e.g., Phase 1, Task 1.1.0, Subtask 1.1.1).
- **Complete one table per task/subtask** covering all seven required fields: objective, approach, location, responsible organization, deliverables, human/animal subjects, and milestones.
- **Name the responsible organization** explicitly (prime, sub-awardee, or consultant) for each task.
- **Specify deliverable IP rights** for any deliverable with less than Unlimited Rights; include the IP assertion in the deliverables cell.
- **State milestones quantitatively.** The template example is explicit: "the assay achieves at least 95% sensitivity and 90% specificity" rather than "the assay has been validated."
- **Cross-walk all tasks and milestones to the Cost Proposal (C.3).** The TDD is the cost cross-walk document; every task must appear in both places.
- **Check the human subjects and animal subjects boxes** at the task level and describe the nature of any such research if applicable.
- **Omit proprietary information** from the TDD; it is not to be included in this document.
- **Align TDD task structure** exactly with the task structure presented in Section 4 (Technical Plan) of Appendix C.1.

## Watch-outs

> [!IMPORTANT]
> The TDD is the primary instrument for post-award milestone tracking and payment justification. Vague milestones directly undermine payment authorization.

> [!IMPORTANT]
> Tasks and milestones in the TDD must cross-walk to the Cost Proposal. Mismatches between C.2 and C.3 are a common source of negotiation delay.

> [!WARNING]
> Qualitative milestone statements ("assay validated," "model trained") are insufficient. The template explicitly requires quantitative performance criteria. ARPA-H reviewers will flag this.

> [!WARNING]
> Do not include proprietary information in the TDD. The document is shared with reviewers and potentially government support contractors.

> [!CAUTION]
> IP assertions must accompany any deliverable with less than Unlimited Rights, listed directly in the Deliverables cell of the task table. Omitting assertions is not the same as claiming Unlimited Rights; it creates ambiguity in negotiations.

> [!CAUTION]
> The human/animal subjects checkbox is at the task level, not just on the cover page. If any subtask involves HSR or ASR, the TDD must flag it and describe the nature of that research.
