# Yar Framework Assessment: The Real Unified App Architecture

**Date:** 2026-07-16
**Scope:** Recommend the long-term app framework for Yar across iOS, Android, Linux, macOS, and Windows, given on-device Gemma-class LLM inference, a low-latency voice loop, and local-first storage with future CRDT sync. Tauri is explicitly the interim desktop build, not the assumed answer.
**Team:** Founder (Shahin) plus one developer (Ali), who built the current Tauri desktop app in React/TypeScript. Existing assets: React/TS frontend, Rust via the Tauri shell, a Django backend being absorbed on-device over time, and a parked legacy Flutter app.

## BLUF

**Primary recommendation: Flutter, paired with `flutter_rust_bridge`, rebuilding the Tauri Rust backend into a shared, UI-agnostic Rust core crate.** This is the only candidate that gives one true UI codebase across all five target platforms (iOS, Android, Linux, macOS, Windows) today, and it currently has the most turnkey packaged story for on-device Gemma (`flutter_gemma`, covering multimodal, audio, and RAG across Android, iOS, and desktop in one dependency). The team already has a parked Flutter app, so this is not a cold start.

**Runner-up: the native-shell-plus-shared-Rust-core pattern** (the Bitwarden/Anytype/Zed model), keeping the existing Tauri and React/TS desktop app unchanged and adding a thin Kotlin Multiplatform and Compose Multiplatform mobile shell over the same Rust core via UniFFI. This scores almost as high and has a higher technical ceiling on AI, voice, and storage performance because there is zero language bridge in the hot path, but it costs more engineering surface for a two-person team (a new mobile UI stack plus custom iOS interop work that is less proven than `flutter_rust_bridge`).

**What would change the call:** if Ali will not take on a second UI language beyond React/TS, drop both of the above and take React Native plus Expo for mobile, keeping the existing Tauri desktop app as-is and sharing a component library between them. This reuses 100 percent of current skills and has a strong, proven local-LLM story (`react-native-executorch`), but it does not satisfy "one codebase" for desktop, since React Native has no credible Linux desktop target.

Either way, Tauri stays as the desktop shell in every scenario. It is being replaced only as the mobile answer.

## Per-Candidate Assessments

**Flutter + `flutter_rust_bridge`.** Flutter is a genuine single Dart codebase across Android, iOS, Linux, macOS, and Windows, with web as a bonus target. On-device Gemma support is the most packaged of any candidate: `flutter_gemma` wraps MediaPipe's LiteRT-LM and covers Gemma 3/4 with multimodal input, function calling, and on-device RAG across Android, iOS, and desktop as a single dependency, with inference and embeddings running through `dart:ffi` directly in the Dart process with no JVM or gRPC hop. `flutter_rust_bridge` is a mature, actively maintained binding generator (a "Flutter Favorite" package) that is the natural vehicle for turning the existing Tauri Rust backend into a shared core; it is already the standard way Flutter apps embed CRDT engines like Loro. Voice latency is workable but requires native plugin code for continuous mic capture, since Dart's isolate model sits between Flutter and the platform audio APIs; this is a solvable problem, not a free one. sqlite-vec has Dart bindings in progress and FTS5 is supported through the standard sqlite3 plugin. The team's own parked legacy Flutter app is a real, if dated, head start. Sources: [flutter_gemma](https://fluttergemma.dev/), [flutter_gemma_litertlm](https://pub.dev/packages/flutter_gemma_litertlm), [Flutter + On-Device AI, 2026](https://thejenildgohel.medium.com/flutter-on-device-ai-running-small-llms-without-a-cloud-backend-2026-edition-4fbd568b585f), [flutter_rust_bridge](https://github.com/fzyzcjy/flutter_rust_bridge), [Rust as a Flutter Data Layer](https://verygood.ventures/blog/rust-as-flutter-data-layer/), [sqlite-vec Flutter PR](https://github.com/asg017/sqlite-vec/pull/119).

**React Native, plus Expo, JSI/TurboModules, `react-native-executorch` or `llama.rn`.** This is the strongest reuse-of-existing-skills option: the team's React/TS component patterns, hooks, and state management transfer with the least friction of any path here. `react-native-executorch` (Software Mansion) is a real, shipping on-device AI toolkit with `useLLM` and `useWhisper` hooks, pre-optimized models on Hugging Face, and reported sub-20ms inference on Snapdragon 8 Elite; Gemma support is arriving on this stack. JSI and TurboModules removed the old bridge's serialization bottleneck, which matters directly for the voice loop. The structural gap is desktop: React Native's desktop story runs through separate, less-maintained forks (`react-native-windows`, `react-native-macos`) with no credible Linux target at all, so this path does not satisfy "one codebase" for desktop; it would keep the existing Tauri/React desktop app as a second, separately maintained surface. Sources: [React Native ExecuTorch docs](https://docs.swmansion.com/react-native-executorch/), [Expo: running AI models with ExecuTorch](https://expo.dev/blog/how-to-run-ai-models-with-react-native-executorch), [React Native On-Device AI, 2026](https://reactnativerelay.com/article/react-native-on-device-ai-run-llms-locally-executorch-react-native-ai).

**Kotlin Multiplatform + Compose Multiplatform.** CMP for iOS went stable in May 2025, and CMP desktop (via Skia/Skiko) genuinely covers Linux, macOS, and Windows, so on paper this is the other real "one codebase everywhere" answer, validated in production at Netflix, McDonald's, Cash App, and JetBrains's own tools. The gap is the AI and interop ecosystem: there is no packaged Gemma library comparable to `flutter_gemma`, so the team would hand-wire `llama.cpp` or MediaPipe through Kotlin/Native cinterop themselves, and connecting a shared Rust core to the iOS target requires manual Kotlin/Native C interop rather than a turnkey generator, since UniFFI's Kotlin bindings target the JVM/Android side first. Neither founder nor Ali has Kotlin or Compose experience today, and this is also the smallest hiring pool of the three cross-platform leaders. Sources: [Is KMP production-ready in 2026?](https://www.kmpship.app/blog/is-kotlin-multiplatform-production-ready-2026), [Is Compose Multiplatform production-ready?](https://volpis.com/blog/is-kotlin-multiplatform-production-ready/).

**Tauri v2 (the current interim build).** Desktop Tauri is genuinely production-grade and is current at the 2.11 line as of July 2026. Mobile is the weak point: Tauri 2.0 added iOS and Android support, but not all official plugins are ported to mobile, and reviewers are direct that "if your product is mobile-first and mobile-only, lean toward React Native or Flutter today." Background execution on mobile is young: the `tauri-plugin-background-service` crate documents iOS's roughly 30-second BGTaskScheduler windows and Android foreground-service requirements, the same constraints every framework faces, but with less mileage behind Tauri's implementation specifically. Its real strength is that the Rust core already embeds SQLite, FTS5, and (with sqlite-vec and Loro added) the full storage and sync stack with zero bridge overhead, which is exactly the asset that should be carried forward into whichever mobile answer is chosen. Sources: [Tauri 2.0 Stable Release](https://v2.tauri.app/blog/tauri-20/), [Build Mobile and Desktop Apps From One Codebase, Tauri 2](https://www.mayhemcode.com/2026/07/build-mobile-and-desktop-apps-from-one.html), [Tauri Mobile Plugin Development](https://v2.tauri.app/develop/plugins/develop-mobile/), [tauri-plugin-background-service](https://crates.io/crates/tauri-plugin-background-service).

**.NET MAUI.** Eliminated. MAUI has no first-class Linux desktop target, which directly fails a stated hard requirement, and Q1 2026 saw regressions serious enough that reviewers flagged it as difficult to run in production during that window. The on-device Gemma and `llama.cpp` ecosystem around C#/.NET is the thinnest of any candidate assessed, and neither team member has .NET experience, so this would mean adopting a new language, a new UI paradigm, and a platform gap simultaneously. Sources: [MAUI vs Avalonia, 2026](https://www.ctco.blog/posts/maui-vs-avalonia-2026-cross-platform-dotnet-ui/), [How are things in 2026?, dotnet/maui discussion](https://github.com/dotnet/maui/discussions/34171).

**Lynx (ByteDance).** Eliminated. Lynx is open source as of early 2025 and already powers parts of TikTok, but desktop support is explicitly "planned," not shipped, there is no found ecosystem for Gemma or `llama.cpp` bindings, and its compiled architecture does not support over-the-air JS bundle updates as of early 2026. One reviewer's framing is apt: "adopting a framework with a one-year history, sparse third-party libraries, and minimal community support for a client-facing production app with a fixed timeline is a bet most teams should not make in 2026." Sources: [Lynx: ByteDance's Blazing Fast JS Framework](https://www.roomacarthur.dev/notes/lynx-bytedances-blazing-fast-javascript-framework-for-cross-platform-apps/), [React Native vs Flutter vs Expo vs Lynx, 2026](https://www.groovyweb.co/blog/react-native-vs-flutter-vs-expo-vs-lynx-2026).

**Capacitor/Ionic.** Eliminated on the voice-loop requirement specifically. Capacitor's native bridge is promise-based, and developers report 20 to 100 milliseconds of latency on each JS-to-native call on Android alone, before any STT or TTS processing happens; forum threads independently confirm background audio is a known pain point. That latency profile conflicts directly with the low-latency bidirectional voice loop this project needs, even though Capacitor otherwise reuses the team's web skills well and has a large hiring pool. Sources: [Native calls have a lot of latency (Android), Ionic Forum](https://forum.ionicframework.com/t/native-calls-have-a-lot-of-latency-android/243664), [Audio in the background for Ionic/Capacitor, Ionic Forum](https://forum.ionicframework.com/t/audio-in-the-background-for-ionic-capacitor-app/234073).

**Native shell per platform, shared Rust core** (the pattern used by Bitwarden's internal SDK, Anytype, and Zed's GPUI). Bitwarden's `bitwarden-core` crate is the single source of truth for business logic across mobile and web, with UniFFI generating Kotlin and Swift bindings from one Rust codebase; Zed runs a shared core with platform-native backends (Metal on macOS, a Win32 backend on Windows) rather than a single cross-platform renderer; 1Password's Rust backend feeds thin, mostly-native frontends per platform. Applied to Yar, this means the existing Tauri Rust backend becomes a standalone engine crate owning SQLite, FTS5, sqlite-vec, Loro, `llama.cpp`/ONNX inference, and the STT/TTS audio pipeline, with zero duplication and zero bridge overhead in the hottest paths (voice and inference), which is why it scores highest of anyone on both those criteria. The existing Tauri/React desktop app needs no rewrite at all; it already is a native shell over this same Rust core. The cost is that mobile needs a real, new UI investment: either genuinely native Swift and Kotlin UIs, or a Compose Multiplatform shell as a lighter-weight substitute, plus bespoke Kotlin/Native cinterop work to reach the Rust core on iOS, since this path lacks a mature, one-command binding generator equivalent to `flutter_rust_bridge`. For a team of one dev, that is real, non-trivial surface area even though the hard logic is centralized. Sources: [Bitwarden SDK architecture](https://contributing.bitwarden.com/architecture/sdk/), [bitwarden_core docs](https://sdk-api-docs.bitwarden.com/bitwarden_core/), [Zed architecture](https://zread.ai/zed-industries/zed/6-zed-vs-vs-code), [Behind the scenes of 1Password for Linux](https://dteare.medium.com/behind-the-scenes-of-1password-for-linux-d59b19143a23).

**On CRDT sync and Go interop specifically:** Loro ships official Rust, JavaScript/WASM, and Swift APIs, and is already used in Flutter apps through `flutter_rust_bridge`, so it fits cleanly under any of the Rust-core-based paths above. Any-sync's Go nodes, by contrast, would need `gomobile` to cross into iOS and Android as XCFrameworks, a separate toolchain from the Rust core with its own build and binary-size overhead; this is a real added cost specifically for the any-sync option versus a Loro-only sync layer, worth weighing separately when `SPEC-sync-protocol.md` is revisited. Sources: [Loro on Hacker News, Flutter integration](https://news.ycombinator.com/item?id=46071730), [gomobile docs](https://pkg.go.dev/golang.org/x/mobile/cmd/gomobile).

## Migration Path from the Interim Tauri Build

1. **Extract the Rust core now, before picking the mobile UI.** Refactor the current Tauri backend so storage (SQLite/FTS5), the future sqlite-vec and Loro layers, and any local LLM inference code live in a standalone Rust crate with no Tauri-specific types in its public API. This work pays off under every option in this document, including staying on Tauri longer.
2. **Prototype both finalists for one to two weeks before committing.** Build a throwaway mobile shell twice: once in Flutter calling the core through `flutter_rust_bridge`, once in Compose Multiplatform calling the core through UniFFI plus manual iOS cinterop. Measure actual voice-loop latency and Gemma load time on a mid-tier Android phone and an iPhone in both. This is cheap relative to the cost of guessing wrong on a two-person team.
3. **Keep the Tauri desktop app running unmodified throughout.** Nothing above requires touching the desktop shell; it continues shipping to Linux, macOS, and Windows users while mobile is built out.
4. **If Flutter wins the prototype:** treat the desktop rewrite in Flutter as a later, lower-urgency step. Ship mobile first on Flutter against the shared core; only fold desktop into Flutter once mobile is stable and the team has bandwidth, since the current Tauri desktop app is not blocking anything in the meantime.
5. **If the native-shell path wins the prototype:** stop there. Desktop needs no further work, and the only remaining build is the Compose Multiplatform (or native Swift/Kotlin) mobile shell against the same core.
6. **Either way, retire Capacitor, Lynx, MAUI, and KMP-without-Flutter as options** rather than revisiting them; the disqualifiers above (Linux gap, immaturity, voice latency) are structural, not tooling version issues that will resolve themselves soon.

## Risks

- **Two-person team, two UI stacks, indefinitely.** Under the native-shell/Rust-core runner-up, the team ends up maintaining React/TS (desktop) and Kotlin/Compose (mobile) simultaneously forever, on top of the Rust core. This is the single biggest reason Flutter is ranked first despite a near-tie score: one Dart UI codebase is less to carry long-term for two people.
- **`flutter_rust_bridge` still needs upkeep as Rust code evolves.** Binding generation is mature but not zero-maintenance; every change to the Rust core's public API requires regenerating and re-verifying bindings on both mobile platforms.
- **On-device Gemma memory ceilings on real phones.** Every candidate here depends on quantized Gemma variants fitting in mobile RAM budgets; none of the frameworks change the underlying model-size math, so this is a model-selection risk independent of the framework choice.
- **iOS background limits cap the "real background operation" requirement everywhere.** BGTaskScheduler's roughly 30-second windows and force-quit kill behavior apply regardless of framework; no candidate here escapes this, it is an iOS platform constraint, not a framework deficiency.
- **Any-sync's Go/gomobile toolchain adds a second cross-compilation pipeline** distinct from whatever handles the Rust core, if any-sync is chosen over a Loro-only sync layer; this is worth resolving in `SPEC-sync-protocol.md` before mobile work starts, not after.
- **Browser extension ("Cytomark") portability differs by choice.** Flutter's web target is real but secondary to native. Keeping the desktop app on Tauri/React keeps the door open to a Cytomark browser extension sharing the existing React/TS component code most directly, an argument in favor of not rushing to move desktop into Flutter even if mobile goes that way.
