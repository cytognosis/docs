# CAP Threat Model

CAP addresses structural authority and accountability. It does not eliminate the need for sandboxing, network controls, model safety evaluation, secure coding, or human governance.

CAP v1 adds explicit Local PEP and Edge PEP enforcement assumptions. A deployment that claims Local PEP bypass resistance must publish its trust mode and attestation or proxy controls. The Therapist/Supervisor scenario is a profile scenario: the Therapist is a local interviewer persona, `raw_data_egress.raw_audio` and `raw_data_egress.raw_transcript` are false by default, and the central/main Supervisor receives only redacted context, evidence references, dimension vectors, and safety flags through a Supervisor Gateway.

| Threat | Attack | CAP mitigation | Remaining gap | Required mitigation |
|---|---|---|---|---|
| Compromised Controller | Controller issues malicious Directives within or beyond its scope. | Holder-bound AuthorityChain warrants bind identity, capability, scope, policy, expiry, revocation metadata, delegation constraints, and signature; GuardDecision and policy refs add independent checks. | Malicious action may be within broad scope. | Least-privilege capabilities; OPA/Cedar scoping; anomaly detection; review of high-risk actions. |
| Compromised Executor | Executor ignores constraints and performs forbidden action. | Refusal/reporting semantics; expected telemetry. | A malicious Executor may lie or omit side effects. | Sandboxing, network egress controls, downstream service authorization, independent auditing. |
| Compromised Guard | Guard allows unsafe Directive. | Multiple Guards and signed decisions possible. | Bad policy or compromised identity can still authorize. | Guard diversity, policy review, key rotation, anomaly detection, independent policy tests. |
| Malicious external A2A agent | External agent accepts task but interprets CAP incorrectly or maliciously. | CAP metadata, authority and evidence semantics. | A2A peer may not enforce CAP. | Require CAP conformance, trust boundaries, signed AgentCard capability declarations, contractual controls. |
| Malicious MCP server | Tool returns poisoned or fabricated results. | Tool result becomes EvidenceRef; evidence treated as untrusted. | Poisoned content may influence Controller. | MCP server authorization, result attestation, evidence sanitization, domain validation. |
| Prompt injection through evidence | Evidence tells model to ignore constraints or exfiltrate data. | Constraints parsed outside model context; EvidenceRef untrusted. | Model may still follow malicious content. | Structured extraction, prompt isolation, output validation, Guard policies on evidence origin/media type. |
| Tool poisoning | Tool implementation replaced with malicious version. | Directive restricts tool id but not binary integrity. | CAP does not attest tool implementation. | SLSA/in-toto supply-chain attestations; MCP server allowlist; runtime integrity checks. |
| Stale evidence | Old evidence reused after context changes. | EvidenceRef freshness and expiry. | Freshness policy may be too weak. | Domain profiles define max age; provenance and snapshots; revalidation before irreversible actions. |
| Stale policy | Policy changes between Guard decision and execution. | PolicyRef version/digest, GuardDecision expiry. | Race conditions remain. | Short GuardDecision TTL; policy revocation checks; deny on policy mismatch. |
| Replay attack | Old Directive resent. | message_id/idempotency_key and expiry. | Clock skew or long TTL can weaken protection. | Short expiry, nonce stores, synchronized clocks, replay caches. |
| Confused deputy | Executor uses its privileges for unauthorized Controller request. | AuthorityChain scopes action; effective constraints. | Downstream systems may trust Executor too broadly. | Per-Directive attenuated credentials; downstream authorization checks; egress policies. |
| Key compromise | Signing key stolen. | Signed authority steps carry expiry and revocation refs; verifiers can reject unknown, revoked, or invalid keys. | Compromised keys can sign malicious steps until rotation/revocation reaches verifiers. | Key rotation, registry-backed revocation freshness, hardware-backed keys, SPIFFE short-lived SVIDs, KMS/HSM integration. |
| Privilege escalation through multi-capability agent | One process acts under broader capability than intended. | Envelope capability fields and AuthorityChainStep capability claims. | Process isolation may be weak. | Capability isolation, separate identities, least privilege, policy engine enforcement. |
| Cross-binding signature confusion | JSON/protobuf transformations break or alter signatures. | CAP v1 helper-generated JSON signatures use RFC 8785/JCS payload bytes and recommend DSSE/COSE/JOSE/in-toto envelopes. Signed objects identify the payload hash, payload type, and canonicalization. | Library misuse or incompatible external v0.1 deterministic JSON migration remains possible. | Use audited libraries; sign typed payloads; test cross-binding transformations and v0.1-to-v1 interop fixtures. |
| Privacy leakage through evidence summary | Summary includes sensitive details unnecessarily. | EvidenceRef confidentiality labels and access policies. | Summaries may leak data despite labels. | Minimization policies, redaction workflows, privacy Guard, telemetry scrubbing. |
| Hidden CoT leakage | Implementation logs scratchpad in DecisionRecord. | DecisionRecord explicitly forbids hidden CoT. | No protocol can inspect all model internals. | Conformance checks, length/structure limits, logging policy, model-provider controls. |
| Audit omission | Observer fails to emit telemetry. | Core conformance requires terminal event telemetry. | Malicious runtime can suppress logs. | Independent collectors, write-once stores, transparency logs, cross-checking downstream systems. |
| Late veto after side effect | Guard denial arrives after action happens. | Lifecycle supports abort/compensation. | Irreversible actions cannot be undone. | Pre-dispatch Guard requirements for irreversible actions; profile-specific human/workflow approval. |
| Evidence access confused with evidence validity | Executor can read evidence but evidence is false. | Hash/freshness only prove identity and time, not truth. | False evidence remains possible. | Source trust policy, multi-source corroboration, domain-specific validation. |
| Local PEP bypass | Therapist/interviewer agent writes directly to user surface, raw-data egress, network, or local tool. | CAP v1 requires profile-declared Local PEP trust mode and one enforced path through the Local PEP for user-visible output, raw evidence egress, tool calls, and network access. The current high-assurance scaffold emits auditable `sidecar_bypass_attempt` refusals for deterministic bypass attempts. | Trusted in-process mode and the current Python scaffold cannot stop malicious code by themselves. | Use separately privileged proxy or attested isolated mode for high-assurance profiles, with OS/platform routing, network egress controls, and local-tool mediation. |
| Supervisor overreach | Central Supervisor requests raw transcript/audio or directs diagnostic, treatment-advice, jurisdiction-violating, or unsafe output. | Local PEP privacy, non-diagnostic, non-prescriptive, jurisdiction, and safety policies can veto Supervisor directives. Current deterministic fixtures keep the Supervisor authority role, Supervisor Gateway endpoint, and backend engine distinct while checking raw-data and unsafe-output refusals against the structured `PrivacyBoundary`. | Bad policies or an incomplete Supervisor Gateway may over-authorize the Supervisor. | Least-privilege supervisor warrants, policy review, anomaly alerts, conformance tests, and a privacy-filtering Supervisor Gateway. |
| Streaming unsafe partial output | Unsafe phrase leaves token buffer before guard fires. | v1 lookahead buffer plus correction-frame semantics. | Very short buffers can miss late-forming unsafe phrases. | Tune buffer by profile, test precision/recall, preserve correction provenance. |
| Control Plane outage | Supervisor Gateway or central PDP is unavailable. | Local PEP applies cached signed policy bundle and offline fallback. | Expired/missing bundle can halt sensitive turns. | Pre-fetch bundles, monitor TTL, fail closed for sensitive profiles. |
| Policy hot-update drift | Local PEP and central PDP apply different policy versions mid-session. | v1 roadmap pins policy bundle per session unless profile forces re-evaluation. | Emergency policy updates may need immediate override. | Define hot-update mode per profile and test both steady-state and forced update behavior. |
| Federated registry stale data | Capability, policy, or evidence metadata is cached past revocation. | Registry refs carry version, digest, freshness, and revocation metadata. | Offline caches trade availability for freshness. | Bound cache TTL, propagate revocations, fail closed for high-risk capabilities. |

## Residual risk statement

CAP conformance means an implementation follows structural authority semantics. It does not mean the system is safe for medical, financial, legal, robotic, or other high-stakes use without an appropriate profile, threat model, independent validation, and operational controls. In the Therapist/Supervisor scenario, CAP can enforce non-diagnostic, non-prescriptive structure and privacy boundaries; it does not certify clinical efficacy or therapeutic quality.
